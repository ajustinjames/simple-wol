# Issue #9 — Run Docker container as non-root user

**Status:** refined
**Branch:** —
**Issue:** https://github.com/ajustinjames/simple-wol/issues/9

## Spec

### Problem

The Dockerfile's runtime stage runs as root. If the application is compromised, the attacker has root access inside the container.

### Implementation

**Modify the runtime stage in `Dockerfile`** to add a non-root user and set data directory ownership:

```dockerfile
FROM alpine:3.19
RUN apk add --no-cache ca-certificates tzdata
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
COPY --from=builder /build/simple-wol /usr/local/bin/simple-wol
RUN mkdir -p /data && chown appuser:appgroup /data
ENV DATA_DIR=/data
EXPOSE 8080
USER appuser
CMD ["simple-wol"]
```

Key changes:
- `addgroup`/`adduser` creates a system user with no login shell
- `mkdir -p /data && chown` ensures the data volume mount point is owned by the app user
- `USER appuser` switches to non-root before CMD

**Files:** `Dockerfile`

### Test Cases

- [x] Container starts successfully as non-root
- [x] Application can create/write to `/data/simple-wol.db`
- [x] `whoami` inside container returns `appuser`, not `root`

## Decisions

1. **System user (`-S`) rather than regular user** — System users have no home directory or login shell by default, which is the correct pattern for service accounts in containers.

2. **Create `/data` dir in Dockerfile rather than relying on volume mount** — If the user doesn't mount a volume, the directory still needs to exist and be writable. The `chown` ensures it works both with and without an external volume.

3. **No `--no-cache` on adduser** — The `apk add` line already uses `--no-cache`. The `adduser`/`addgroup` commands don't use the package cache.
