# Issue #6 — Add request body size limits

**Status:** refined
**Branch:** —
**Issue:** https://github.com/ajustinjames/simple-wol/issues/6

## Spec

### Problem

All JSON body parsing uses `json.NewDecoder(r.Body).Decode()` with no size limit. An attacker can send multi-gigabyte request bodies to exhaust server memory.

### Implementation

**Add a helper function `limitBody`** in `main.go`:
```go
func limitBody(w http.ResponseWriter, r *http.Request, maxBytes int64) {
    r.Body = http.MaxBytesReader(w, r.Body, maxBytes)
}
```

**Call `limitBody` at the top of every handler that decodes JSON:**
- `handleSetup` (main.go:134) — 1KB is plenty for username/password
- `handleLogin` (main.go:165) — 1KB
- `handleCreateDevice` (main.go:280) — 1KB
- `handleUpdateDevice` (main.go:302) — 1KB

Use a const like `maxRequestBody = 1024` (1KB). Device and auth payloads are tiny JSON objects — 1KB is generous.

**Handle the `MaxBytesError`**: When `json.Decode` fails after `MaxBytesReader` truncates, the error message includes "http: request body too large". Check for this in the existing error handling or return the generic 400 error — the current `"invalid request"` response is already appropriate.

**Files:** `main.go`

### Test Cases

- [x] Request body under limit → accepted normally
- [x] Request body over limit → 400 error response
- [x] Exact boundary (1024 bytes) → accepted
- [x] Just over boundary (1025 bytes) → rejected

## Decisions

1. **1KB limit rather than 1MB** — The largest payload this app handles is a device object (~200 bytes of JSON). 1KB provides 5x headroom while still protecting against abuse. The audit suggested "1MB or less" but there's no reason to allow that much.

2. **Inline `MaxBytesReader` calls over a middleware** — A body-limiting middleware would apply to GET requests (which have no body) and couldn't vary limits per endpoint. Inline calls are explicit and allow per-endpoint tuning if needed later.

3. **No special error message for size exceeded** — The existing `"invalid request"` 400 response is sufficient. The client doesn't need to know the specific reason the decode failed.
