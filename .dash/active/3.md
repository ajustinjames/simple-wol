# Issue #3 — Enforce password strength requirements

**Status:** refined
**Branch:** —
**Issue:** https://github.com/ajustinjames/simple-wol/issues/3

## Spec

### Problem

`CreateUser()` in `auth.go:26` passes the password directly to `bcrypt.GenerateFromPassword` with no validation. The `handleSetup` handler in `main.go:134-163` only checks that the password is non-empty (`req.Password == ""`). This means:

1. Single-character passwords are accepted
2. Passwords longer than 72 bytes are silently truncated by bcrypt — the user believes they have a 100-char password but only the first 72 bytes are verified on login

### Implementation

**Add a `ValidatePassword` function in `auth.go`** that enforces:
- Minimum 8 characters
- Maximum 72 bytes (bcrypt limit) — reject rather than silently truncate
- Return a descriptive error message for each violation

**Call `ValidatePassword` in `handleSetup`** (main.go:148) before calling `CreateUser`. Return 400 with a JSON error describing the requirement that failed.

**Files:** `auth.go`, `main.go`

### Test Cases

- [ ] Password shorter than 8 characters → rejected with clear message
- [ ] Password exactly 8 characters → accepted
- [ ] Password at 72 bytes → accepted
- [ ] Password exceeding 72 bytes → rejected with clear message
- [ ] Empty password → rejected (existing behavior preserved)
- [ ] Multi-byte UTF-8 password where char count is under 72 but byte count exceeds 72 → rejected

## Decisions

1. **Reject passwords over 72 bytes rather than warn** — A warning would require UI changes and the user might ignore it. Hard rejection is simpler and prevents the truncation surprise entirely.

2. **No complexity requirements (uppercase, special chars, etc.)** — NIST SP 800-63B recommends against composition rules. Length is the primary strength factor. This keeps the UX simple for a single-user home tool.

3. **Validation in a standalone function rather than inline** — Makes it testable independently and reusable if a password-change endpoint is added later.
