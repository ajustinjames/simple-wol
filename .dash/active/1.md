# Issue #1 — UI/UX fixes: button sizing, scan issue, and setup flow feedback

**Status:** refined
**Branch:** fix/ui-ux-improvements
**Issue:** https://github.com/ajustinjames/simple-wol/issues/1

## Spec

### 1. Add Device button too wide on desktop

**Problem:** `.btn-primary` has `width: 100%` (style.css:67), which makes every primary button full-width. In the toolbar context on desktop, this stretches the "Add Device" button across the entire row.

**Fix:** Remove `width: 100%` from `.btn-primary` (it's only needed on the login page). Apply `width: 100%` scoped to `.login-container .btn-primary` instead. The toolbar buttons already use `btn-sm` and will auto-size to their content.

**Files:** `static/style.css`

### 2. No confirmation after initial account creation

**Problem:** After successful setup, `handleSetup()` in `login.html:78` redirects to `/login` silently. The user lands on the login form with no feedback that their account was created.

**Fix:** After successful `/api/setup` response, show a success message (e.g., "Account created. Please log in.") on the login form. Approach: redirect to `/login?setup=success` and show a styled confirmation banner when the query param is present.

**Files:** `static/login.html`

### 3. Network scan lacks progress feedback

**Problem:** The scan works correctly but the UI shows a static "Scanning network..." message with no progress indication. On larger subnets this can take several seconds, making it appear stuck.

**Fix:** Add a CSS spinner animation to the "Scanning network..." loading state. Show error message on failure or render results on success. No API changes — purely frontend.

**Files:** `static/app.js`, `static/style.css`

### 4. Buttons too small for mobile

**Problem:** `.btn-sm` has `padding: 0.25rem 0.5rem` and `font-size: 0.75rem` (style.css:82-85). On mobile, this results in touch targets well below the recommended 44px minimum.

**Fix:** Add a media query for mobile (max-width: 600px) that increases `.btn-sm` padding and font-size to meet the 44px tap target threshold. Target: `min-height: 44px; padding: 0.5rem 0.75rem; font-size: 0.875rem`.

**Files:** `static/style.css`

## Design

### Button sizing (desktop toolbar)
- "Add Device" and "Scan Network" buttons in toolbar should be auto-width (content-sized), not full-width
- Login/setup page buttons remain full-width within the centered form container

### Mobile touch targets
- All `.btn-sm` elements get minimum 44px height on viewports <= 600px
- Increase padding and font-size proportionally

### Setup confirmation
- Green success banner appears at the top of the login form: "Account created successfully. Please log in."
- Uses a `.success` CSS class (green-tinted variant of the existing `.error` style)
- Banner is shown by detecting `?setup=success` query parameter on page load

## Decisions

1. **Scoped `width: 100%` to login container rather than removing `.btn-primary` entirely** — Other future primary buttons outside the toolbar may also want auto-width. Scoping to `.login-container` is the minimal change.

2. **Query param approach for setup confirmation vs. session flash message** — Query param (`?setup=success`) is simpler and doesn't require server-side session flash logic. Downside: URL looks slightly ugly, but it's transient (user logs in immediately).

3. **CSS spinner over SSE for scan progress** — SSE would give real progress (e.g., "pinged 120/254 hosts") but adds significant complexity. An animated spinner with elapsed time is sufficient to communicate "still working" and requires only frontend changes.
