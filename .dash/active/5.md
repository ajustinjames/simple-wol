# Issue #5 — Add CSRF protection to state-changing endpoints

**Status:** refined
**Branch:** —
**Issue:** https://github.com/ajustinjames/simple-wol/issues/5

## Spec

### Problem

State-changing endpoints (POST, PUT, DELETE) rely solely on session cookies. While `SameSite=Strict` provides partial protection, it's not a complete CSRF defense.

### Implementation

**Use the custom header pattern** — require an `X-Requested-With: XMLHttpRequest` header on all state-changing API requests. This is the simplest CSRF defense for a JSON-only API:

- Cross-origin form submissions cannot set custom headers
- `SameSite=Strict` already blocks cross-site cookie attachment
- The combination provides defense in depth

**Add validation in `requireAuth` middleware (`main.go:112-130`):** For non-GET requests, check that `r.Header.Get("X-Requested-With") == "XMLHttpRequest"`. Return 403 with `{"error":"forbidden"}` if missing.

**Update `static/app.js`:** Add the `X-Requested-With: XMLHttpRequest` header to the `api()` helper function for all requests.

**Update `static/login.html` (or `login.js` after issue #4):** Add the header to the `handleSetup` and `handleLogin` fetch calls. Note: `/api/setup` and `/api/login` are not behind `requireAuth`, so this is defense-in-depth rather than required.

**Files:** `main.go`, `static/app.js`, `static/login.html` (or `static/login.js`)

### Test Cases

- [x] POST/PUT/DELETE to protected endpoints without `X-Requested-With` header → 403
- [x] POST/PUT/DELETE with correct `X-Requested-With` header → allowed (normal auth flow)
- [x] GET requests work without the custom header
- [x] Frontend adds header to all mutating requests
- [x] Login and setup still work (they bypass `requireAuth`)

## Decisions

1. **Custom header pattern over synchronizer tokens** — This app is a single-page JSON API with no form submissions. Synchronizer tokens add significant complexity (token generation, storage, injection into pages) for no additional benefit in this architecture. The custom header approach is recommended by OWASP for this pattern.

2. **Validation in `requireAuth` rather than a separate middleware** — All state-changing endpoints already go through `requireAuth`. Adding the check there avoids a second middleware layer and ensures coverage.

3. **`X-Requested-With` over a custom `X-CSRF-Token`** — Both work equally well for preventing cross-origin requests. `X-Requested-With` is a well-known convention and doesn't require token management.
