# Issue #10 — Add TLS/HTTPS support or document reverse proxy requirement

**Status:** refined
**Branch:** —
**Issue:** https://github.com/ajustinjames/simple-wol/issues/10

## Spec

### Problem

The server only listens on plain HTTP. Passwords and session cookies are transmitted in cleartext. The `Secure` cookie flag is conditional on `X-Forwarded-Proto`, but there's no documentation about deploying behind a reverse proxy.

### Implementation

**Document the reverse proxy requirement rather than adding native TLS.** Native TLS adds certificate management complexity (paths, renewal, Let's Encrypt integration) that is inappropriate for a simple home tool. Users deploying this will typically already have Caddy, nginx, or Traefik handling TLS for their home lab.

**Add a "Deployment" section to `README.md`** covering:
- The application listens on HTTP (default port 8080)
- For production, place behind a TLS-terminating reverse proxy
- Example Caddy and nginx snippets
- The `Secure` cookie flag is automatically set when `X-Forwarded-Proto: https` is present

**Add `TLS_CERT` and `TLS_KEY` env vars for optional native TLS** as a lightweight addition:
- If both are set, use `http.ListenAndServeTLS` instead of `http.ListenAndServe`
- If only one is set, log an error and exit
- Log which mode the server is using at startup

**Files:** `main.go`, `README.md`

### Test Cases

- [ ] Without TLS env vars → server starts on HTTP (existing behavior)
- [ ] With both `TLS_CERT` and `TLS_KEY` → server starts on HTTPS
- [ ] With only one TLS var set → error and exit
- [ ] `Secure` cookie flag set when using native TLS

## Decisions

1. **Documentation-first approach with optional native TLS** — Pure documentation would leave users who want TLS without a reverse proxy unsupported. Optional env-var-based TLS is minimal code (~10 lines) and covers both use cases.

2. **No HTTP-to-HTTPS redirect** — Adding a redirect requires listening on two ports or a redirect middleware. For a home tool this is unnecessary complexity — the user will configure their reverse proxy or bookmark the HTTPS URL.

3. **No Let's Encrypt / ACME integration** — This would require a domain name, DNS configuration, and adds significant dependency weight. Out of scope for a LAN tool.
