# Issue #16 — Add CI/CD pipeline for testing and in-place update releases

**Status:** refined
**Branch:** —
**Issue:** https://github.com/ajustinjames/simple-wol/issues/16

## Spec

### CI: Test Workflow
A GitHub Actions workflow that runs `go test ./...` on every PR and push to main. This complements the existing CodeQL workflow (`.github/workflows/codeql.yml`) by catching functional regressions alongside security analysis.

### CD: Release Pipeline
Two-stage release triggered by PR labels:

1. **Version bump workflow** (`.github/workflows/version-bump.yml`) — Runs on merge to main. Reads the PR's `major`, `minor`, or `patch` label, fetches the latest `v*.*.*` tag, calculates the next version, and creates + pushes the new tag. PRs without a semver label are skipped (no release).
2. **Release workflow** (`.github/workflows/release.yml`) — Triggered by the new tag. GoReleaser cross-compiles binaries for linux/amd64 and linux/arm64, builds a Docker image, and publishes everything as a GitHub Release with checksums.

**Semver strategy:** PR-label-based. Label a PR with `major`, `minor`, or `patch` before merging. The automation handles tag creation — no manual `git tag` needed. PRs merged without a semver label produce no release.

**Costs:** Zero. GitHub Actions is free for public repos (2,000 minutes/month for private). GoReleaser OSS edition is free. No external services required.

### In-Place Updates
Users running bare-metal installations can update via a curl one-liner that downloads the latest release binary from GitHub Releases. A `/api/version` endpoint reports the running version so users (or a future UI banner) can detect when updates are available.

## Decisions

1. **PR-label-based semver over manual tagging** — Labels (`major`, `minor`, `patch`) on PRs let the maintainer declare intent at review time. A version-bump workflow reads the label on merge, calculates the next tag, and pushes it. No manual `git tag` needed. PRs without a label skip the release — safe by default.

2. **GoReleaser over manual `go build` matrix in Actions** — GoReleaser handles cross-compilation, checksums, changelog generation, and GitHub Release publishing in a single config file. The alternative is ~50 lines of custom workflow YAML that reimplements the same thing.

3. **`/api/version` endpoint over embedded version in binary name** — An API endpoint enables programmatic update checks. Version is injected at build time via `-ldflags` which GoReleaser handles natively.

4. **linux/amd64 + linux/arm64 only** — Covers x86 servers and Raspberry Pi / ARM home servers. Darwin/Windows users will use Docker. Adding more platforms later is a one-line GoReleaser config change.

5. **No self-update mechanism in the binary** — A curl one-liner is sufficient for a home tool. Embedding update logic adds dependency weight, security surface (signature verification), and complexity disproportionate to the use case.

6. **Docker image published via GoReleaser, not a separate workflow** — Keeps release artifacts unified. GoReleaser's Docker support handles multi-arch images and tagging (`:latest`, `:v1.0.0`).

## Test Cases

- [ ] PR triggers test workflow and reports pass/fail status check
- [ ] Push to main triggers test workflow
- [ ] Merging a PR labeled `patch` creates the correct next tag (e.g., v1.0.0 → v1.0.1)
- [ ] Merging a PR labeled `minor` bumps minor and resets patch (e.g., v1.0.1 → v1.1.0)
- [ ] Merging a PR labeled `major` bumps major and resets minor/patch (e.g., v1.1.0 → v2.0.0)
- [ ] Merging a PR with no semver label does not create a tag or release
- [ ] Created tag triggers release workflow
- [ ] Release creates GitHub Release with linux/amd64 and linux/arm64 binaries
- [ ] Release includes SHA256 checksums
- [ ] `/api/version` returns the version injected at build time
- [ ] `/api/version` returns `"dev"` when running without `-ldflags`
- [ ] Existing Dockerfile still builds correctly
- [ ] GoReleaser Docker image builds and is pushed
