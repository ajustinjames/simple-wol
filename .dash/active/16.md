# Issue #16 — Add CI/CD pipeline for testing and Docker releases

**Status:** refined
**Branch:** feature/setup-cicd
**Issue:** https://github.com/ajustinjames/simple-wol/issues/16

## Spec

### CI: Test Workflow
A GitHub Actions workflow that runs `go test ./...` on every PR and push to main. This complements the existing CodeQL workflow (`.github/workflows/codeql.yml`) by catching functional regressions alongside security analysis.

### CD: Release Pipeline
Two-stage release triggered by PR labels:

1. **Version bump workflow** (`.github/workflows/version-bump.yml`) — Runs on merge to main. Reads the PR's `major`, `minor`, or `patch` label, fetches the latest `v*.*.*` tag, calculates the next version, and creates + pushes the new tag. PRs without a semver label are skipped (no release).
2. **Release workflow** (`.github/workflows/release.yml`) — Triggered by the new tag. Builds a multi-arch Docker image (linux/amd64, linux/arm64) via `docker/build-push-action`, pushes to GHCR with version and `latest` tags, and creates a GitHub Release with auto-generated changelog.

**Semver strategy:** PR-label-based. Label a PR with `major`, `minor`, or `patch` before merging. The automation handles tag creation — no manual `git tag` needed. PRs merged without a semver label produce no release.

**Costs:** Zero. GitHub Actions is free for public repos (2,000 minutes/month for private). No external services required.

### Version Endpoint
A `/api/version` endpoint reports the running version so users (or a future UI banner) can detect when updates are available. Version is injected at build time via `-ldflags`; defaults to `"dev"` for local builds.

### Updating
Docker is the sole distribution method. Users update via `docker compose pull && docker compose up -d`.

## Decisions

1. **PR-label-based semver over manual tagging** — Labels (`major`, `minor`, `patch`) on PRs let the maintainer declare intent at review time. A version-bump workflow reads the label on merge, calculates the next tag, and pushes it. No manual `git tag` needed. PRs without a label skip the release — safe by default.

2. **Docker-only release over GoReleaser binary distribution** — GoReleaser adds config complexity (`.goreleaser.yml`), a third-party tool dependency, and produces binary artifacts that need their own install/update instructions. Docker is already the primary deployment method for this project. A single `docker/build-push-action` step is simpler and sufficient. Users update with `docker compose pull` — no curl one-liners or binary management needed.

3. **`/api/version` endpoint over embedded version in binary name** — An API endpoint enables programmatic update checks. Version is injected at build time via `-ldflags` in the Dockerfile.

4. **linux/amd64 + linux/arm64 only** — Covers x86 servers and Raspberry Pi / ARM home servers. These are the two relevant platforms for a home LAN tool.

5. **No self-update mechanism** — `docker compose pull` is the update path. No binary distribution means no need for curl scripts, checksums, or signature verification.
